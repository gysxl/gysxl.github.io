闪瞎眼健身app概要设计

发表于 Jun 23 2017

需求分析

整个app提供四个主要功能：登录以及权限管理、打卡积分、查看打卡记录、积分兑换。

登录以及权限管理功能需求列表

  • 健身团成员以手机号为登录账号，通过密码登录，或者通过验证码登录方式，名称使用真实姓名，便于管理员confirm申请。
  • 普通团员角色，登录后可正常查看打卡记录，发起当天打卡申请，查看本人积分，查看本人兑换记录的功能。
  • 管理员除了普通成员的功能外，还有confirm成员打卡申请的权限。

打卡积分功能

  • 成员正常登陆后，界面显示当月的日历表格，当前日期高亮，底部显示积分数目。
  • 若人员当日还未打卡，则打卡健身按钮高亮，日历当日正常显示，点击打卡健身按钮后，按钮置灰，文案显示为“等待管理员确认”（每天控制打卡次数为1次）；
  • 若当日用户已打卡成功，则打卡健身按钮置灰，若此时管理员没有confirm此次打卡，则显示“等待管理员确认。。。”；
  • 若当日已成功打卡，且管理员已经确认，则日历中当天日期显示小星星一枚。
  • 管理员确认打卡申请后，当前用户积分加一。

管理员确认功能

  • 当前成员登录后，若匹配后台为管理员角色，则轮询comfirm接口，判断是否有待确认打卡记录。
  • 若有未confirm请求，则以列表形式将待confirm信息现在在界面，主要显示用户名称，管理员选中需要确认的用户打卡申请后，通过点击列表底部的确认打卡按钮，confirm成员的打卡记录。

积分兑换功能

  • 用户正常登陆后，在界面底部会显示用户当前积分，若积分大于等于5分，则对应的积分换取礼品按钮高亮。
  • 点击“换取礼品”按钮后，跳转至积分商城页面
  • 积分商城界面根据用户当前积分，显示可换取礼品（可选取礼品可点击，积分不足礼品置灰）
  • 用户选中某礼品后，点击“确认兑换”，用户积分相应减去该礼品所需积分，并返回打卡首页。

接口设计

login接口

  • 功能
    1、登录app
    2、获取用户权限
  • 调用时机：用户登录app时
  • 数据提交方式：POST
    1、传参信息：
参数名称 参数说明 参数类型 是否必传 默认值
phone 用户手机号 string 是 无

2、正确返回值：

返回参数说明

参数名称 参数说明 备注
UID 用户id 数据库中生成
roles 用户角色 0：管理员 1：普通用户

search接口

  • 功能
    1、查询历史打卡数据
    2、查询当天是否已提交打卡申请
    3、查询现有积分数据
  • 调用时机：用户登录成功后
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
UID 用户id string 是 无
month 查询日期 string 是 当前日期

2、正确返回值

返回参数说明

参数名称 参数说明 备注
history_list 用户签到历史数据 查询签到数据表获取
current_date 当前用户今天是否已提交签到申请 0：未提交 1：已提交
coupon_now 当前用户所剩积分 查询积分表

push 打卡接口

  • 功能
    1、用户提交打卡申请
  • 请求时机：用户当天没有提交给申请前提下，点击打卡按钮
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
UID 用户id string 是 无
data 打卡日期 string 是 当前日期

2、正确返回值

  • 备注
    1、申请记录存入redis缓存中，key值为当日日期，value为UID，多个成员申请，则以“,”分隔

search_push 查询成员打卡申请接口

  • 功能
    1、管理员登录后查询成员今日打卡申请记录
  • 调用时机：管理员登录成功后
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
data 查询日期 string 是 当前日期

2、正确返回

返回参数说明

参数名称 参数说明 备注
pushs 当日提交打卡申请用户信息 uid与姓名

confirm_push 管理员确认打卡接口

  • 功能
    1、管理员确认成员打卡成功
    2、管理员确认成员打卡失败
    3、对应申请成员积分+1
    4、清除成员申请时提交的redis数据
  • 调用时机：管理员点击“确认打卡”按钮
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
data 确认日期 string 是 当前日期
uid_list 确认打卡用户列表 string 是 无

2、正确返回

备注：该接口主要用来将redis中的请求删除，并将打卡成功的成员填入数据库，并且积分数据+1

convert 兑换礼品接口

  • 功能
    1、用户提交兑换礼品申请
    2、用户积分减少
  • 请求时机：用户选中某个礼品，点击“确认兑换”按钮
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
present_id 兑换礼品id string 是 无

2、正确返回值

search_convert 查询成员兑换申请接口

  • 功能
    1、管理员查看当前礼品申请信息
  • 请求时机：管理员点击查看“申请信息”按钮
  • 数据提交方式：POST，无参数
    1、正确返回值

返回参数说明：

参数名称 参数说明 备注
presents 当前未处理的兑换列表 礼品与兑换人信息

confirm_convert 确认礼品已下发接口

  • 功能
    1、礼品兑换发放确认
  • 调用时机：管理员点击“确认发放”按钮
  • 数据提交方式：POST
    1、传参信息
参数名称 参数说明 参数类型 是否必传 默认值
presents 成员申请记录（礼品id，用户id） string 是 无

2、正确返回值

数据库设计

整个模块共计需要5张数据表，以及一个redis缓冲池，详细设计如下：
1、用户信息表
2、用户打卡记录表
3、礼品信息表
4、兑换记录表
5、用户积分表
